# HDMIエンコーダの関数分割・共通化およびコンパイラの構造体マッピング対応

## 1. 背景と課題

`src/hdmi/encoder/encoder.cm` では、DVI 1.0 に準拠した 8b/10b エンコード処理が Red, Green, Blue の3つのチャネルそれぞれに対してコピペで記述されています。これにより、中間変数の assign 宣言や DC バランス制御の分岐（計約 220 行）が重複し、コードの保守性や見通しが損なわれていました。

これらの重複コードを共通の純粋関数に抽出し、役割ごとに整理します。
また、将来的な構造体による抽象化をサポートするため、Cmコンパイラの SystemVerilog コード生成部（`codegen.cpp`）が構造体型（`TypeKind::Struct`）の変数宣言時に正しい型名を出力できない問題を修正します。

## 2. 設計と方針

### 2.1 共通関数の抽出

エンコーダ処理を以下の4つの純粋関数（状態に依存しない組み合わせロジック）に分割します。

1. `uint get_qm(uint din)`:
   - 8-bitのピクセルデータ入力から、遷移最小化処理を適用した 9-bit の中間コードを算出します。
2. `uint get_n1_qm(uint qm)`:
   - 9-bit 中間コード内の `1` の個数を数えます。
3. `uint get_tmds(uint qm, uint n1_qm, int cnt, bool de, uint ctrl)`:
   - 累積不均衡 `cnt` およびコントロールトークン `ctrl`（同期信号用など）に基づき、最終出力となる 10-bit の TMDS コードを決定します。
4. `int get_next_cnt(uint qm, uint n1_qm, int cnt, bool de)`:
   - 更新された新しい累積不均衡カウンタ（ディスパリティ）を算出して返します。

### 2.2 状態の安全な引き渡し
ハードウェア（SystemVerilog）で合成可能で、かつコンパイラがサポートする範囲で安全に状態を更新するため、ポインタなどの参照を使わず、値を受け取って更新後の値を返すピュアな関数として定義します。
`tmds_encode` 関数内では、各チャネルのローカルな累積レジスタ（`cnt_r`, `cnt_g`, `cnt_b`）に対して、`get_next_cnt` の戻り値を代入して毎サイクル状態を更新します。

### 2.3 Cmコンパイラ側の型マッピング対応
`Cm/src/codegen/sv/codegen.cpp` の `SVCodeGen::mapType` を修正し、`hir::TypeKind::Struct` に対面した際に `type->name` を返すように変更します。これにより、SVターゲット向けの変数や引数宣言で構造体名が正しく出力されるようになります。

## 3. リファクタリング後のエンコーダ構造

```
[ピクセルデータ (din)]
     │
     ▼
[get_qm] (遷移最小化)
     │
     ├───► [get_n1_qm] (1の個数カウント)
     │          │
     ▼          ▼
[get_tmds]  [get_next_cnt] (DCバランス計算)
     │          │
     ▼          ▼
[10b TMDS]  [次サイクル cnt]
```
