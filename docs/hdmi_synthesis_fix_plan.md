# HDMI 合成エラーの解決方針およびビルドスクリプト改善設計

## 1. Gowin Synthesis (macOS) における合成クラッシュの解決方針

### 1.1 問題の背景
macOS版の Gowin Synthesis ツールにおいて、クロック同期プロセス内のブロッキング代入や、モジュール内部での `function` 宣言を含んだコードを処理しようとすると、最適化フェーズ（Optimizing Phase 1）の直後に `libc++abi: terminating due to uncaught exception of type std::length_error: basic_string` の例外がスローされ、プロセスがクラッシュする不具合が発生していました。

### 1.2 解決のアプローチ
- **関数の完全インライン展開**: `lookup_font` などの論理を、呼び出し元のクロック同期プロセス `render_process` にすべてインライン化しました。これに伴い、`font_rom.cm` は不要になります。
- **純粋な非ブロッキング・パイプライン構造へのリファクタリング**: 逐次依存関係のあった論理を、明示的な3ステージのパイプラインレジスタ（非ブロッキング代入 `<=` のみ）に書き換えることで、ツール側での一時変数の最適化クラッシュを防ぎます。
- **インポートのクリーンアップ**: `hdmi_text_top.cm` の `import ./text/font_rom;` は不要になったため削除します。
- **テスト・本番用ソースパスの復元**: `gowin_hdmi.tcl` でデバッグ用に切り替えていた `hdmi_colorbar_7df0eb2.sv` へのパスを、元の `hdmi_colorbar.sv` に戻します。

## 2. `builder.sh` の対話型プロジェクト選択機能の改善設計

### 2.1 現状の課題
複数の `.cm` ソースファイル（例: `main.cm`, `hdmi_text_top.cm`）が存在するディレクトリで `./builder.sh` を引数なしで起動した場合、対話型プロンプトによりプロジェクトの選択を求められます。しかし、現状はインデックス値（1, 2など）の数値入力しか受け付けておらず、文字列で直接プロジェクト名（`hdmi_text_top` 等）を指定することができません。

### 2.2 改善後のロジック
選択プロンプトの読み取りロジック（`choice`）を以下のように拡張します：
1. 入力値が数値であり、かつ選択肢の範囲内である場合はインデックス参照によりプロジェクトソースを決定。
2. 入力値が数値でない場合、検出された全プロジェクトファイル（`cm_files`）のベース名と完全一致するものがあるかを検索し、一致するものがあればそれを選択。
3. いずれも一致しない場合は無効な入力としてエラー終了。

これにより、CUI環境での自動化や直接的な名前入力がスムーズに行えるようになります。
