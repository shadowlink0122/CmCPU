# Gowin EDA 合成器クラッシュ（std::length_error）の修正設計書

## 1. 現象と原因分析

`./builder.sh hdmi --apply` を実行した際、Gowin EDA の論理合成フェーズにおいて以下のエラーが発生してビルドが失敗します。

```
[15%] Optimizing Phase 1 completed
libc++abi: terminating due to uncaught exception of type std::length_error: basic_string
[ERROR] Gowin EDA 合成失敗
```

### 原因の深層分析
一時変数宣言（`_tXXXX`）を削減した後も同様のクラッシュが発生し、精査した結果、生成された SystemVerilog コードにおける**32ビット暗黙的型拡張およびシフト量バレルシフタの超巨大化**が原因と判明しました。

1. **32ビット定数による暗黙的幅拡張**:
   Cmコード内の `1` や `2` などの定数がすべて `32'd1` や `32'd2` などの 32ビット幅指定定数として SV 上に出力されていました。
   このため、`r_out & 32'd1` などの演算がすべて 32ビット幅で評価され、さらにこれらが深くネストした複雑な論理ツリーとなっていたため、論理合成時の計算量が指数関数的に爆発していました。

2. **シフト量の32ビットバレルシフタ推論**:
   `r_out >> 32'd1` のように、シフト量（右オペランド）に 32ビット幅の定数が渡されると、合成器は32ビット幅対応のバレルシフタを推論しようとします。
   この32ビットのシフタが3チャネル分の popcount などで合計20箇所以上も推論されるため、GWS 最適化器の処理能力限界を超えて `std::length_error` を引き起こしていました。

---

## 2. 対策方針

### ① HDMIソースコードの変数型最適化（実施済み）
`src/hdmi/pattern/pattern.cm` および `src/hdmi/encoder/encoder.cm` 内の変数の型を 32ビット `int` から、適切なビット幅（8ビット `utiny`/`tiny`、16ビット `short`）に変更します。
これにより、元となる回路の最大ビット幅が 16ビットに抑えられます。

### ② SVバックエンド (`codegen.cpp`) の定数生成の最適化（今回追加）
定数リテラルの出力および二項演算生成を以下のように改善します。

1. **定数ビット幅の強制縮小制御** (`codegen.cpp` 309行目付近):
   `target_width` （相手オペランドのビット幅）が指定されている場合、元のリテラルの明示幅（Cm 上での 32ビット `int`）に関わらず、定数リテラルのビット幅を `target_width` に縮小して出力するようにします（例: `r_out & 32'd1` → `r_out & 8'd1`）。

2. **シフト量の幅なし定数化** (`codegen.cpp` 452行目付近):
   二項演算がシフト演算（`<<`, `>>`）の場合、右辺のシフト量（定数リテラルの場合）はビット幅指定なしの単純な整数リテラル（例: `1`, `2` 等）として直接出力するようにします（例: `r_out >> 32'd1` → `r_out >> 1`）。

## 3. 動作検証計画

1. **Cmコンパイラのビルド**:
   ```bash
   make -C Cm build
   ```
2. **HDMIのビルドとコード確認**:
   ```bash
   ./builder.sh hdmi
   ```
   * `build/hdmi/hdmi_colorbar.sv` 内の `always_ff` ブロックにおける演算が `r_out & 8'd1` や `r_out >> 1` に変わっていることを確認します。
3. **実機デプロイ**:
   ```bash
   ./builder.sh hdmi --apply
   ```
   * Gowin EDA の合成および P&R がクラッシュせず正常終了し、実機への書き込みが完了することを確認します。
