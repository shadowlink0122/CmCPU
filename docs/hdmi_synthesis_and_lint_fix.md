# HDMIカラーバー回路の合成フォールバックおよびリント警告の修正設計

## 1. Gowin EDA デバイスフォールバック設計

### 背景
Sipeed Tang Console 138K 実機に搭載されている FPGA パッケージは 676ピンの `FPG676` (`GW5AST-LV138FPG676AC2/I1`) です。しかし、ローカルの開発マシンにインストールされている Gowin EDA のライセンスやデータベースによっては 676ピンパッケージが登録されておらず、プロジェクトの作成時にデバイスが見つからないエラーとなります。

一方で、484ピンパッケージの `GW5AST-LV138PG484AC1/I0` を指定するとプロジェクト作成は成功しますが、ピン制約ファイル（`.cst`）に記述された `AA22` や `AA23` などのピンが存在しないため、配置配線（P&R）ステップでエラーになります。

### 設計
Gowin EDA Tcl スクリプト（`gowin_hdmi.tcl`）において、以下の動的判別・制御フローを実装します。

1. **デバイスサポート判定**:
   まず一時的なディレクトリで `GW5AST-LV138FPG676AC2/I1` を指定したプロジェクトの作成を試みます。
2. **フォールバック処理**:
   - 失敗した場合（デバイス未対応環境）:
     デバイスを `GW5AST-LV138PG484AC1/I0` に変更し、ピン制約ファイル（`.cst`）の追加をスキップした上で、**合成のみ（`run syn`）** を実行します。これにより、ピンアサインエラーを回避してコードの論理合成が正常に通るかをチェックします。
   - 成功した場合（デバイス対応環境）:
     通常通り `run all` を実行し、合成、P&R、ビットストリーム（`.fs`）生成までを一括して実行します。

## 2. Verilator リント警告の解消設計

### 背景
Cm からコンパイルされた SystemVerilog コードにおいて、変数の型と数値リテラルのビット幅のミスマッチが原因で Verilator から多くの幅拡張（`WIDTHEXPAND`）および切り詰め（`WIDTHTRUNC`）警告が発生しています。

- `r_out` (8bit) に対する `32'd255` (32bit) の代入。
- `r_q0` 〜 `r_q7` (32bit) を使用したシフト・論理演算および 16bit の `tmds_r` への代入。

### 設計
Cm コンパイラの lexer/parser は `8'd255` や `16'd852` などのビット幅指定リテラルをサポートしているため、Cm ソースコード側でこれを明示的に指定します。

1. **型定義の修正**:
   `tmds_encode` 内で使用する一時的なビット変数 `r_q0` 〜 `r_q7`、`g_q0` 〜 `g_q7`、`b_q0` 〜 `b_q7` の型を `uint`（32bit）から `ushort`（16bit）に変更します。
2. **キャストと幅付きリテラルの利用**:
   - `r_out = 8'd255;` のように8ビットリテラルを使用。
   - 8ビットの `r_out` をシフトして1ビットを抽出する際、`ushort`（16bit）にキャストして計算を行います。
     ```rust
     r_q1 = (((r_out as ushort) >> 16'd1) & 16'd1) ^ r_q0;
     ```
   - 最終的な `tmds_r` の合成処理も16ビットリテラルと16ビット変数のみで行います。
     ```rust
     tmds_r = r_q0 | (r_q1 << 16'd1) | ... | (16'd1 << 16'd8);
     ```

これにより、SystemVerilog 生成コード内のすべての演算が同一ビット幅で揃い、Verilator リントチェックでの警告数が 0 になります。
